# SDD Orchestrator & AI Agents Ecosystem

Добро пожаловать в экосистему ИИ-разработки на базе **Spec-Driven Development (SDD)**. 

### Шаг 1: Установка
Склонируйте этот скилл в папку `.agents/skills/sdd-orchestrator`.
Для работы Дизайнера требуется зависимый скилл `frontend-design`. Выполните команду для его автоматической установки:
```bash
python .agents/skills/sdd-orchestrator/scripts/install_deps.py
```

Этот скилл (`sdd-orchestrator`) превращает вашу среду разработки в фабрику по производству качественного кода. Вместо того чтобы просить ИИ "просто написать приложение", вы запускаете строго оркестрованный процесс, где архитектура, спецификации и сам код создаются по строгим правилам (SOLID, TDD, Value Objects).

---

## 🛠️ Как подключить агентов к новому проекту?

Чтобы Оркестратор и агенты-исполнители могли начать работу в вашем новом репозитории, **вам, как человеку-пользователю, необходимо выполнить два шага**:

### Шаг 1: Настройка глобальных правил (AGENTS.md)
Скопируйте или создайте файл `.agents/AGENTS.md` в корне вашего проекта. В нем должны быть описаны базовые поведенческие правила для ИИ (чтобы он общался на русском, делал git-коммиты и использовал методологию SDD).
*Убедитесь, что там есть строка-отсылка на вашу архитектуру:*
`Стек технологий и архитектура определены в файле .openspec/system-architecture.md`.

### Шаг 2: Определение стека технологий (System Architecture)
Создайте папку `.openspec/` в корне проекта и добавьте туда файл `system-architecture.md`. 
Вам нужно **жестко зафиксировать** в этом файле, какие технологии будут использоваться в проекте:
*   **Frontend**: (например, Angular 18+, Tailwind, RxJS)
*   **Backend**: (например, NestJS, TypeORM, PostgreSQL)
*   **DevOps**: (например, Linux VPS, Docker, GitHub Actions)
*   **Embedded**: (например, ESP32, ESP-IDF, C++)

*Без этого файла агенты не смогут понять, на каком языке им писать код, так как скиллы намеренно сделаны независимыми от фреймворков (framework-agnostic).*

## 🚀 Как ставить задачи агентам?

В методологии SDD ИИ-агенты **никогда не пишут логику без спецификаций**. 

1. **Создание фичи с нуля**:
   Вы пишете Оркестратору команду: *"Спроектируй новый модуль корзины покупок"*. 
   Оркестратор создаст `proposal.md`, `design.md` и локальные `specs.md`. Только после того, как вы (человек) одобряете эти Markdown-спецификации, Оркестратор дает команду писать код.
2. **API-first подход**:
   Прежде чем Backend-разработчик напишет хоть строчку серверного кода, в папке `.openspec/` должен лежать согласованный `api-contract.yaml` (OpenAPI 3.0). ИИ-бэкендер работает строго по этому контракту.

## 🤖 Роли агентов (Кто есть кто)

В нашей системе ИИ разделен на строгие профили (персоны), которые лежат в папке `resources/`. Оркестратор сам выбирает нужную персону для конкретной задачи:

*   👑 **Оркестратор (SDD-Orchestrator)**: Архитектор системы. Не пишет реализацию. Пишет спецификации, проектирует API-контракты и проверяет код, написанный другими агентами, на соответствие правилам (SOLID).
*   🎨 **UI-Designer**: Специалист по визуалу. Читает `specs.md` и генерирует статичную верстку (HTML/CSS) и дизайн-токены с помощью скилла `frontend-design`. Запрещено писать логику.
*   ⚛️ **Frontend-Engineer**: Разработчик логики. Берет готовую верстку от UI-Designer, "натягивает" ее на компоненты (Angular/React), подключает RxJS/стейт и интегрирует API. Запрещено менять дизайн.
*   ⚙️ **Backend-Worker**: Специалист по серверу. Пишет контроллеры, сервисы и БД-миграции строго по `api-contract.yaml`. Запрещено писать HTML/CSS.
*   🛠️ **DevOps-Engineer**: Настраивает Docker, CI/CD и безопасность Linux VPS. Придерживается принципов Infrastructure as Code (IaC).
*   🤖 **Embedded-Worker**: Специалист по микроконтроллерам (ESP32/STM32). Работает с железом, прерываниями и FreeRTOS. Запрещено трогать веб-логику.

**Готовы начать?** Настройте `system-architecture.md` и просто напишите в чат: *"Изучи архитектуру проекта и спроектируй новый модуль [название]"*.
