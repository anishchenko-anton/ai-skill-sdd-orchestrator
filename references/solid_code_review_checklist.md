# SOLID and Clean Code Review Checklist

This checklist is used by Orchestrators to audit code generated by Worker subagents during Step 6 (Validation).

---

## 1. Single Responsibility Principle (SRP)

- **Class Check**: Does this class do exactly one thing? (e.g. A service should fetch data, but should *not* parse routing parameters or format HTML strings).
- **Component Check**: UI component classes (e.g., in Angular, React, Vue) should strictly govern the view logic. Push any business logic, API requests, or local storage persistence into dedicated Services or Hooks.

---

## 2. Open-Closed Principle (OCP)

- **Extensibility**: Can you add new functionality without modifying the existing class? 
- **Example Check**: Instead of writing nested `switch` cases or chains of `if/else` on type flags, use polymorphism, strategy patterns, or lookup tables.

---

## 3. Liskov Substitution Principle (LSP)

- **Inheritance**: Subclasses must be substitutable for their base classes without breaking correctness.
- **Example Check**: Avoid overriding methods in derived classes to throw `NotImplementedError` or return values that violate the parent class interface contract.

---

## 4. Interface Segregation Principle (ISP)

- **Lean Interfaces**: Classes should not be forced to depend on interface methods they do not use.
- **Example Check**: Prefer small, cohesive interfaces (e.g., `interface TaskDeleter { delete(id: string): void }`) over large, monolithic ones.

---

## 5. Dependency Inversion Principle (DIP)

- **Abstraction Dependency**: High-level modules should not depend on low-level modules; both must depend on abstractions.
- **Example Check**: Always inject services using dependency injection (e.g. constructor injection in UI Frameworks or Backend services) instead of hardcoding instances with `new ServiceName()`.

---

## 6. Clean Code Review Rules

- **Intention-Revealing Names**: Never use short names like `d` or `t`. Use `elapsedTimeInDays` or `createdTask`.
- **Early Returns (No Else Blocks)**: Eliminate nested code structures. Return immediately on invalid state checks to keep code linear and readable.
  * *Incorrect*:
    ```typescript
    if (user) {
      if (user.isActive) {
        return user.name;
      } else {
        return "Inactive";
      }
    } else {
      return "Guest";
    }
    ```
  * *Correct*:
    ```typescript
    if (!user) return "Guest";
    if (!user.isActive) return "Inactive";
    return user.name;
    ```
- **Length Limits**: Functions must be **< 20 lines** of code. Classes must be **< 50 lines** of code.
